# Encode — Study Environment + Structured Knowledge Vault

## Vision

Encode is a desktop study app that makes you think harder about what you're learning. It's built on one insight from learning science: the bottleneck is never how much you consume — it's how deeply you process what you consumed. Every other study app makes consumption easier. Encode makes digestion unavoidable.

At its core, Encode is two things:

1. **A study environment** that imports content, forces you to stop and think before continuing, and connects new material to everything you've previously learned.
2. **A structured markdown vault** that stores all your knowledge as plain text files — readable anywhere, editable with any tool, and usable as context for any AI (local or cloud).

The vault format is the product. The app is one interface to it, not the only one. Your notes work in Obsidian, your flashcards can be generated by Claude Code CLI, your quizzes can come from Gemini CLI or a local Ollama model. Nothing is locked in.

### Grounded in

**Learning science** (Dr. Justin Sung's research):
- Three Pillars: Enablers → Retrieval → Encoding
- PACER: Consume then digest — always balanced
- Bloom's Taxonomy: Push thinking up from recall to evaluation/creation
- Desirable difficulty: Learning should feel challenging — that's it working

**Habit formation research:**
- Implementation intentions ("if coffee, then mind-map one concept")
- Simple cue-behavior pairing beats motivation every time
- 18-254 days to form a habit (mean: 66 days)

**Unix philosophy:**
- Plain text files as the universal interface
- Do the study environment well, let other tools handle the rest
- Composable with CLI tools, scripts, other apps

## The 7 MVP Features

### 1. One-Click Web Import

**What:** A browser bookmarklet that sends the current page URL to Encode. The app fetches it, runs Mozilla Readability to strip it to clean content, and saves it as a well-structured markdown file in your vault.

**Why it matters:** Getting content into the app has to be frictionless. One click. If it takes more than that, people will just keep reading in the browser and never engage the digestion features.

**How it works:**
```
Browser → Bookmarklet (1 click) → Encode local HTTP server (port 7878)
→ Fetch page → Readability extraction → Chunk into sections
→ Save as markdown file with YAML frontmatter → Open in reader
```

The bookmarklet is ~1 line of JavaScript:
```javascript
javascript:void(fetch('http://localhost:7878/import',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({url:location.href})}))
```

PDF import also supported — drag and drop a PDF, app extracts text and converts to markdown.

The resulting markdown file:
```markdown
---
subject: D426 Data Management
topic: Normalization - Second Normal Form
source_url: https://wgu.edu/course/d426/chapter3
imported_at: 2026-03-21T19:00:00
word_count: 2340
estimated_read_minutes: 12
status: unread
---

# Second Normal Form (2NF)

## Section 1: What is 2NF?

A relation is in 2NF if it is in 1NF and every non-key
attribute is fully functionally dependent on the entire
primary key...

## Section 2: Partial Dependencies

A partial dependency occurs when a non-key attribute
depends on only part of a composite primary key...

[content continues, chunked by headings]
```

### 2. Reader with Digestion Gates

**What:** A focused reading pane that renders your imported markdown content section by section. After each section, a digestion gate blocks you from continuing until you engage with the material.

**Why it matters:** This is the centerpiece of the entire app. It directly attacks the core problem — people consume hours of content and retain almost nothing because they never stop to process. The gate forces the stop. It creates the consumption/digestion balance that Justin Sung identifies as the most neglected part of learning.

**How the gate works:**

After each content section, the reader shows a prompt. The prompt type rotates:

- **Summarize:** "Explain what you just read in one sentence."
- **Connect:** "How does this relate to something you already know?"
- **Predict:** "What do you think comes next and why?"
- **Apply:** "Where in real life could you use this?"

You type a response. If AI is available (Ollama locally or Claude API), it evaluates whether the response shows actual understanding or is a throwaway to skip past. If no AI is available, the gate still works — you still have to type something, and the act of articulating forces processing.

Your response gets appended to the markdown file:

```markdown
## Digestion: Section 1
**Prompt:** Explain what you just read in one sentence.
**Response:** 2NF removes columns that only depend on part of
a composite key, forcing them into their own table.
**AI Feedback:** Accurate. You captured the core idea. Consider:
what happens to queries that previously joined on the original
table? *(2026-03-21 7:12pm)*
```

**Consumption:Digestion ratio** shown as a simple bar at the top of the reader. Green (balanced), amber (2:1), red (3:1+). No complex dashboard — just one indicator.

### 3. Encoding Coach

**What:** A chat interface where AI asks you questions designed to push your thinking up Bloom's taxonomy. It pulls from your entire vault — old notes, old analogies, old quiz failures — to make questions targeted and personal.

**Why it matters:** This is the feature that makes new learning build on old learning. When you're struggling with 3NF, the coach doesn't ask generic questions. It finds your 2NF notes, your GasMart analogy, your quiz failure on transitive dependencies, and uses all of that to scaffold the new concept onto what you already know.

**How it pulls old knowledge:**

```
1. User is reading about 3NF
2. App runs full-text search across vault: "normalization"
   "2NF" "dependency" "normal form" within same subject folder
3. Finds: 2NF notes with GasMart analogy, quiz where user
   failed on "compare 2NF vs 3NF", teach-back transcript
   where user explained 2NF clearly
4. All of this becomes context in the AI prompt
5. AI asks: "You connected 2NF to fuel price duplication at
   your stores. 3NF goes further — if you store employee
   city AND zip code, and city comes from zip code, that's
   a transitive dependency. How is this different from the
   partial dependency problem you already understand?"
```

The vault search is simple — SQLite FTS5 indexes all markdown content. The search query pulls relevant excerpts. Those excerpts become prompt context. Whether it's Ollama or Claude doing the thinking, they both get the same rich context from your vault.

**AI is optional.** Without AI, the encoding coach doesn't work. But the app gracefully shows: "Enable Ollama or add a Claude API key to activate the encoding coach." The reader and digestion gates still function.

### 4. Bloom's Quiz Generator

**What:** AI generates quizzes at specific Bloom's taxonomy levels from your markdown notes. Adapts difficulty based on your history.

**How it generates quizzes:**

The app reads one or more markdown files from your vault, feeds the content to the AI (local Ollama or cloud API), and requests questions at specific levels.

For a local Ollama model (Llama 3.1 8B, Mistral 7B, Phi-3):
- Levels 1-3 (Remember, Understand, Apply): Good quality. Multiple choice with plausible distractors.
- Level 4 (Analyze): Decent. Compare/contrast questions work.
- Levels 5-6 (Evaluate, Create): Weak. These require nuance that small models struggle with.

For Claude/Gemini API:
- All levels work well. Open-ended evaluation is where cloud AI shines.

**Adaptive behavior:** Quiz distribution targets one Bloom's level above what you've been demonstrating. The quiz results are appended to the topic's markdown file and also saved as their own file:

```markdown
---
subject: D426 Data Management
topic: Normalization
quiz_date: 2026-03-21
ai_model: ollama/llama3.1:8b
---

# Quiz: Normalization (2026-03-21)

## Question 1 — Bloom's Level 2 (Understand)
**Q:** Explain in your own words why partial dependencies cause problems.
**Your Answer:** Because if data depends on only part of the key,
you end up duplicating it across rows, which means updates can
make data inconsistent.
**Assessment:** Correct. You identified both the duplication and
the inconsistency consequence. ✓

## Question 2 — Bloom's Level 4 (Analyze)
**Q:** Compare how 2NF and 3NF each address data redundancy.
What type of redundancy does each eliminate?
**Your Answer:** 2NF handles when data depends on part of the
key. 3NF handles... I'm not sure.
**Assessment:** You understand 2NF well but 3NF is unclear.
3NF eliminates transitive dependencies — when a non-key
attribute depends on another non-key attribute. ✗

## Summary
- Level 2: ✓ (strong)
- Level 4: ✗ (gap: transitive dependencies)
- **Next session focus:** 3NF transitive dependencies
```

**CLI alternative:** You can skip the in-app quiz entirely and just run:
```bash
claude "Read ~/Encode/subjects/D426/chapters/03-normalization.md
and quiz me with 5 questions, Bloom's levels 3-5"
```
The markdown files work with any tool.

### 5. Flashcard Review

**What:** Flashcards stored as markdown files with spaced repetition metadata in YAML frontmatter. The app reads them, presents due cards, and updates the file after review.

**Two sources of flashcards:**

1. **You create them** (encoding) — While studying, you hit a shortcut to create a card from the current content. You decide what goes on front and back. This is encoding work.

2. **AI generates them** (convenience) — Local Ollama or cloud API reads your notes and generates cards. You review and keep/edit/delete. The generation saves you busywork but the review still forces engagement.

**Flashcard file format:**

```markdown
---
subject: D426 Data Management
topic: Normalization
type: flashcards
---

# Flashcards: Normalization

---

> [!card] id: fc-001
> **Q:** What problem does 2NF solve that 1NF doesn't?
> **A:** Partial dependencies — when a non-key attribute depends
> on only PART of a composite key, not the whole key.
> **Bloom:** 2
> **Tags:** #2NF #partial-dependency
> **Ease:** 2.5
> **Interval:** 7
> **Next:** 2026-04-04
> **Last:** 2026-03-28

---

> [!card] id: fc-002
> **Q:** Your GasMart store table has (StoreID, ProductID) as
> composite key and includes City. Why does this violate 2NF?
> **A:** City depends only on StoreID, not on the full composite
> key (StoreID, ProductID). It's a partial dependency.
> **Bloom:** 3
> **Tags:** #2NF #real-world
> **Ease:** 2.8
> **Interval:** 14
> **Next:** 2026-04-11
> **Last:** 2026-03-28

---
```

**Review flow:** App scans all flashcard files in the subject folder, finds cards where `Next` date ≤ today, presents them in a review session. After each card, you rate recall (Again / Hard / Good / Easy), the app updates the card's spaced repetition fields in the markdown file.

**CLI generation:**
```bash
# Generate flashcards from your notes using any AI
ollama run llama3.1 "Read this and generate 10 flashcards
in the exact format shown. Use Bloom's levels 2-4.
$(cat ~/Encode/subjects/D426/chapters/03-normalization.md)"

# Or use Claude Code
claude "Read all md files in ~/Encode/subjects/D426/chapters/
and generate a flashcard file for each chapter"
```

### 6. Feynman Explain-Back (Text)

**What:** You select a topic and type an explanation as if teaching someone with no background. AI evaluates the depth, accuracy, and gaps.

**Why text, not audio (for MVP):** Recording, transcription, and audio playback are three separate engineering challenges. Typing your explanation gets 90% of the Feynman benefit — the act of articulating forces you to discover what you don't actually understand. Audio is Phase 2.

**How it works:**

1. Select a topic
2. App shows: "Explain [topic] as if teaching someone who knows nothing about it. Use simple language. If you use an analogy, explain why it fits."
3. You type 3-10 sentences
4. AI evaluates: simplicity, accuracy, completeness, analogy quality, gaps found
5. Result saved as markdown:

```markdown
---
subject: D426 Data Management
topic: Second Normal Form
type: teach-back
date: 2026-03-21
score: 7
---

# Teach-Back: Second Normal Form (2026-03-21)

## My Explanation
Imagine you have a spreadsheet tracking what products each store
sells. The row ID is store+product combined. But you also put
the store's city in every row. The city only cares about the
store, not the product. So if the store moves cities, you'd
have to update hundreds of rows. 2NF says take that city column
out and put it in its own store table. Now city is in one place.

## AI Evaluation
- **Simplicity:** 9/10 — Clear, no jargon
- **Accuracy:** 8/10 — Correct core concept
- **Completeness:** 6/10 — Didn't mention that 2NF requires
  1NF first, or define "partial dependency" explicitly
- **Analogy:** Strong — the spreadsheet example works well
- **Gaps:** prerequisite relationship to 1NF, formal definition
  of partial dependency
- **Concepts you understand well:** redundancy, update anomalies
- **Work on next:** formal dependency terminology

## Comparison Note
*First teach-back on this topic. Future attempts will show
improvement over time.*
```

### 7. One Thing Daily Commitment

**What:** The app opens to one question every day: "What is the one encoding action you're committing to today?" You set a cue ("after my coffee") and a behavior ("mind-map one concept from Chapter 4"). App tracks if you did it.

**Why it matters:** This doesn't directly teach anything. It builds the habit of showing up. Without this, every other feature is useless because the app stays closed.

**Stored as markdown:**

```markdown
---
type: daily
date: 2026-03-21
---

# Daily Commitment: 2026-03-21

**Cue:** After my morning coffee
**Action:** Read and digest one section of D426 Chapter 3
**Completed:** true
**Completed at:** 7:45pm
**Reflection:** Got through 2NF section. The digestion gate on
partial dependencies made me realize I didn't actually understand
composite keys as well as I thought. Made a flashcard for it.
```

**Streak tracking:** App counts consecutive days with completed commitments. Displayed simply — no gamification, no badges, just "Day 12."

### Vault Rules

1. **Every piece of knowledge is a `.md` file.** No knowledge lives only in SQLite.
2. **YAML frontmatter on every file.** Subject, topic, type, dates — enables filtering and search.
3. **Wiki-links between files.** `[[03-normalization]]` links chapters. The app and Obsidian both resolve these.
4. **Mermaid for visual maps.** No separate canvas — diagrams live inside markdown, rendered by the app.
5. **Flashcard SR metadata in the file.** Ease, interval, next review — all in the card block. Readable by humans, parseable by the app.
6. **SQLite is just an index.** Delete `encode.db` and the app rebuilds it from the markdown files on next launch. Zero data loss.

## SQLite Schema (Index + Scheduling Only)

```sql
-- Full-text search index (rebuilt from vault files)
CREATE VIRTUAL TABLE vault_fts USING fts5(
  file_path,
  subject,
  topic,
  content,
  tokenize='porter'
);

-- Spaced repetition schedule (derived from flashcard files but
-- cached here for fast "what's due today?" queries)
CREATE TABLE sr_schedule (
  card_id TEXT PRIMARY KEY,       -- e.g., "normalization-cards/fc-001"
  file_path TEXT NOT NULL,
  next_review DATE NOT NULL,
  interval_days REAL DEFAULT 1.0,
  ease_factor REAL DEFAULT 2.5,
  last_reviewed DATE
);

-- Daily commitment streak tracking
CREATE TABLE daily_streaks (
  date DATE PRIMARY KEY,
  commitment_text TEXT,
  completed BOOLEAN DEFAULT FALSE,
  completed_at DATETIME
);

-- Quiz score history (for adaptive difficulty)
CREATE TABLE quiz_history (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  subject TEXT NOT NULL,
  topic TEXT NOT NULL,
  bloom_level INTEGER NOT NULL,
  correct BOOLEAN NOT NULL,
  attempted_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- File index (for fast lookups without scanning disk)
CREATE TABLE file_index (
  file_path TEXT PRIMARY KEY,
  subject TEXT,
  topic TEXT,
  file_type TEXT,   -- 'chapter', 'flashcard', 'quiz', 'teach-back', 'map', 'daily', 'capture'
  word_count INTEGER,
  updated_at DATETIME,
  frontmatter TEXT  -- cached YAML as JSON
);
```

## AI Integration Architecture

### Tiered AI — Local First, Cloud Optional

```
┌─────────────────────────────────────────────┐
│                  Encode App                   │
│                                               │
│  ┌──────────────┐    ┌──────────────────┐    │
│  │  AI Request   │───▶│  AI Router       │    │
│  │  (quiz, coach │    │                  │    │
│  │   evaluate)   │    │  1. Try Ollama   │    │
│  └──────────────┘    │  2. Fall back to  │    │
│                       │     Claude API   │    │
│                       │  3. Fall back to  │    │
│                       │     Gemini API   │    │
│                       │  4. No AI mode   │    │
│                       └──────────────────┘    │
└─────────────────────────────────────────────┘
```

**The AI Router** tries providers in order based on user config:

1. **Ollama (default):** Free, local, private. Good for Bloom's 1-4 quizzes, flashcard generation, basic digestion gate evaluation. App checks `http://localhost:11434/api/tags` on startup to detect Ollama.

2. **Claude API (optional):** Set API key in settings. Used for encoding coach (Bloom's 5-6), deep Feynman evaluation, nuanced digestion gate feedback. The app batches context efficiently — one call with the full relevant vault context, not many small calls.

3. **Gemini API (optional):** Alternative to Claude. Same capabilities, different provider.

4. **No AI mode:** The reader and digestion gates still work — you type summaries but don't get evaluation feedback. Flashcards work — you create and review them manually. Quiz and encoding coach are disabled. The app is still useful as a structured study environment with a markdown vault.

### What Each AI Does

| Feature | Ollama (free) | Claude/Gemini (paid) | No AI |
|---------|--------------|---------------------|-------|
| Digestion gate prompt | ✓ Shows prompt | ✓ Shows prompt | ✓ Shows prompt |
| Gate response evaluation | Basic (accept/retry) | Nuanced feedback | Skipped (always accept) |
| Flashcard generation | ✓ Good for Bloom 1-3 | ✓ Good for all levels | Manual only |
| Quiz generation L1-3 | ✓ Good | ✓ Good | Not available |
| Quiz generation L4-6 | Decent | ✓ Strong | Not available |
| Open-ended answer eval | Basic | ✓ Deep, nuanced | Not available |
| Encoding coach | Passable | ✓ Best experience | Not available |
| Feynman evaluation | Basic | ✓ Detailed scoring | Not available |
| Vault search | ✓ (SQLite FTS5, no AI needed) | ✓ | ✓ |

## UI Design

1. **Dark mode first** — Reduces eye strain for long sessions
2. **One Thing as home screen** — App opens to daily commitment, not a dashboard
3. **Minimal chrome** — Content and your thinking are the focus
4. **Sidebar navigation** — Subjects, vault browser, quick access to review/quiz
5. **Reader is split-pane** — Content top, digestion gate bottom (not side-by-side — vertical split keeps reading width comfortable)
6. **Keyboard-first** — Shortcuts for everything: next section, open gate, create flashcard, start quiz
7. **No gamification** — No points, badges, XP, leaderboards. Just honest data: streak count, cards reviewed, gates completed
8. **Calming palette** — Dark bg, muted purples and teals, high-contrast text
9. **Mermaid renders inline** — Maps display visually in the reader, edit as text in markdown
